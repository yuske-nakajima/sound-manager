# コードレビュースキル

## 目的
実装されたコードの品質、セキュリティ、パフォーマンス、保守性を確認する。

## レビュー観点

### 1. セキュリティ
- [ ] パスワードやAPIキーがハードコーディングされていないか
- [ ] ユーザー入力の検証が適切に行われているか
- [ ] XSS、SQLインジェクションなどの脆弱性がないか
- [ ] 認証・認可が適切に実装されているか
- [ ] 機密情報がログに出力されていないか
- [ ] データ削除時にユーザー確認が取られているか

### 2. パフォーマンス
- [ ] 不要なレンダリングが発生していないか（React）
- [ ] 無駄なAPI呼び出しがないか
- [ ] 大量データ処理が効率的か
- [ ] メモ化（useMemo、useCallback）が適切に使用されているか
- [ ] N+1問題が発生していないか
- [ ] 適切なインデックスが使用されているか（DB）

### 3. 可読性・保守性
- [ ] 変数名、関数名が分かりやすいか
- [ ] コメントが適切に付けられているか（必要な箇所のみ）
- [ ] ファイルサイズが200行以内か（超える場合は分割が必要）
- [ ] 責務が適切に分離されているか
- [ ] 複雑な処理が関数に分割されているか
- [ ] マジックナンバーが定数化されているか

### 4. テスト
- [ ] テストが書かれているか
- [ ] 正常系のテストがあるか
- [ ] 異常系のテストがあるか
- [ ] 境界値のテストがあるか
- [ ] テストカバレッジが適切か
- [ ] 既存のテストが全て通るか

### 5. 型安全性（TypeScript）
- [ ] any が使用されていないか
- [ ] interface / type が適切に定義されているか
- [ ] 型定義は interface、合成は type alias が使われているか
- [ ] オプショナル（?）が適切に使用されているか
- [ ] 型ガードが必要な箇所で使用されているか

### 6. エラーハンドリング
- [ ] try-catch が適切に使用されているか
- [ ] エラーメッセージが分かりやすいか
- [ ] エラー時の処理が適切か
- [ ] エラーログが出力されているか
- [ ] ユーザーにエラーが適切に通知されるか

### 7. 設計原則への準拠
- [ ] YAGNI（不要な機能がないか）
- [ ] DRY（重複コードがないか）
- [ ] KISS（シンプルに保たれているか）
- [ ] SOLID（設計原則に従っているか）
  - 単一責任の原則
  - 開放/閉鎖の原則
  - リスコフの置換原則
  - インターフェース分離の原則
  - 依存性逆転の原則

### 8. コーディング規約
- [ ] `references/code-style.md` に準拠しているか
- [ ] インポート順序が正しいか
- [ ] 命名規則に従っているか
- [ ] インデント、改行が統一されているか

## 成果物の作成

### レビュー結果の保存

レビューを実施した場合は、以下のディレクトリに結果を保存します：

1. **ディレクトリの確認・作成**
   ```bash
   mkdir -p .tmp/reviews
   ```
   - ディレクトリが存在しない場合は必ず作成する

2. **レビュー結果の保存先**
   - `.tmp/reviews/YYYY-MM-DD_<レビュー対象>.md`
   - 例: `.tmp/reviews/2025-12-19_pr-123-review.md`

3. **レビュー結果の内容**
   - レビュー日時
   - レビュー対象（PR番号、ブランチ名など）
   - 指摘事項（Critical/Major/Minor）
   - 良かった点
   - 総評（LGTM / 修正が必要 / 議論が必要）

4. **ユーザーへの報告**
   - 作成したファイルのパスを報告
   - レビュー結果の要約を提示

## レビュープロセス

### 1. 変更内容の確認
- 何を実装したのか理解する
- 変更ファイル一覧を確認
- 変更の影響範囲を把握

### 2. コードの静的解析
- Serena MCP を使ってコード構造を把握
  - `mcp__serena__get_symbols_overview` でファイル構造確認
  - `mcp__serena__find_symbol` でシンボル詳細確認
  - `mcp__serena__find_referencing_symbols` で影響範囲確認

### 3. チェックリストに沿ったレビュー
- 上記のレビュー観点に従って確認
- 問題があれば具体的な指摘と修正案を提示

### 4. テストの実行
- 既存テストが通ることを確認
- 新規テストが適切に書かれているか確認

### 5. フィードバック
- 問題点を明確に指摘
- VSCode などでジャンプできるようにリンクを貼る
  - 例: `app/services/user.ts:42`
- 修正案を具体的に提示
- 良い点も伝える（ポジティブフィードバック）

## 指摘のレベル

### Critical（必須修正）
- セキュリティリスク
- データ損失の可能性
- 本番環境でのエラー
- パフォーマンス問題（致命的）

### Major（修正推奨）
- 設計原則違反
- テスト不足
- 型安全性の問題
- エラーハンドリング不足

### Minor（改善提案）
- 命名の改善
- コメント追加
- リファクタリング提案
- パフォーマンス改善（軽微）

## レビューコメントの例

### Good（具体的な指摘）
```
❌ app/services/user.ts:42
パスワードがハードコーディングされています。環境変数を使用してください。

修正案:
- const password = 'secret123';
+ const password = process.env.DB_PASSWORD;
```

### Bad（不明瞭な指摘）
```
セキュリティ問題があります。
```

## レビュー後のアクション

### 問題なし
- ✅ LGTM（Looks Good To Me）を伝える
- マージ可能と判断

### 修正が必要
- 具体的な修正内容を伝える
- 修正後に再レビュー

### 議論が必要
- 設計の妥当性など判断が難しい場合
- ユーザーに確認を求める

## ツール活用

### Serena MCP
- `mcp__serena__get_symbols_overview`: コード構造把握
- `mcp__serena__find_symbol`: シンボル詳細確認
- `mcp__serena__find_referencing_symbols`: 影響範囲確認

### その他
- `Read`: ファイル内容確認
- `Grep`: パターン検索
- `Bash`: テスト実行

## 関連スキル
- `implementation`: 実装ガイドラインの参照
- `planning`: 計画との整合性確認
