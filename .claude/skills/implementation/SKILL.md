# 実装ガイドライン

## 技術スタック
- **言語**: TypeScript
- **フレームワーク**: React
- **テスト**: 必須（テストフレームワークはプロジェクトに準拠）
- **スタイル**: プロジェクトの既存スタイルに従う

## 開発原則

### TDD（Test-Driven Development）
- t_wada の TDD で開発する
- Red → Green → Refactor のサイクルを守る

### 設計原則
- **YAGNI**: You Aren't Gonna Need It - 不要な機能は実装しない
- **DRY**: Don't Repeat Yourself - 重複を避ける
- **KISS**: Keep It Simple, Stupid - シンプルに保つ
- **SOLID**: オブジェクト指向設計の5原則
  - Single Responsibility Principle（単一責任の原則）
  - Open/Closed Principle（開放/閉鎖の原則）
  - Liskov Substitution Principle（リスコフの置換原則）
  - Interface Segregation Principle（インターフェース分離の原則）
  - Dependency Inversion Principle（依存性逆転の原則）

## コーディング規約

詳細は `references/code-style.md` を参照。主なルール：

### TypeScript
- **型定義**: interface を使う
- **合成**: type alias を使う
- **any の禁止**: 基本的に any は使わない
- 型安全性を最優先

### ファイル分割
- 1つのファイルがおおよそ **200行を超える** ときはファイル分割する
- 責務ごとに適切に分割

### エラーハンドリング
- **必須**: すべての実装でエラーハンドリングを実装
- try-catch を適切に使用
- ユーザーに分かりやすいエラーメッセージ

### セキュリティ
- **NEVER**: パスワードやAPIキーをハードコーディングしない
- **NEVER**: ユーザーの確認なしにデータを削除しない
- セキュリティベストプラクティスに従う
- XSS、SQLインジェクションなどの脆弱性に注意

### パフォーマンス
- パフォーマンスへの影響を常に考慮
- 不要なレンダリング、API呼び出しを避ける

## 成果物の管理

### 調査結果・設計ドキュメントの作成

実装前に調査や設計が必要な場合は、以下のディレクトリに成果物を保存します：

1. **ディレクトリの確認・作成**
   ```bash
   mkdir -p .tmp/research
   mkdir -p .tmp/design
   ```
   - ディレクトリが存在しない場合は必ず作成する

2. **調査結果の保存先**
   - `.tmp/research/YYYY-MM-DD_<調査内容>.md`
   - 例: `.tmp/research/2025-12-19_api-integration-research.md`

3. **設計ドキュメントの保存先**
   - `.tmp/design/YYYY-MM-DD_<設計内容>.md`
   - 例: `.tmp/design/2025-12-19_database-schema-design.md`

4. **ユーザーへの報告**
   - 作成したファイルのパスを報告
   - ファイルの内容の要約を提示

## 実装フロー

### 1. インターフェース定義から開始
- まず型定義・インターフェースを定義
- 関数のシグネチャを先に決める

### 2. テストケース作成
- 実装前にテストを書く（TDD）
- 正常系、異常系、境界値をカバー

### 3. 実装
- 小さい単位で実装
- 1コミット = 1つの論理的な変更
- 既存のテストが通ることを確認してから変更

### 4. リファクタリング
- マーティン・ファウラーが推奨するリファクタリング手法に従う
- テストが通る状態を保ちながらリファクタリング
- コードの可読性、保守性を向上

## 必須チェック項目

実装完了前に以下を確認：

- [ ] 型安全性が保たれているか
- [ ] any を使用していないか
- [ ] エラーハンドリングが実装されているか
- [ ] テストが書かれているか
- [ ] 既存のテストが全て通るか
- [ ] セキュリティリスクがないか
- [ ] パフォーマンスへの影響を考慮したか
- [ ] ファイルサイズが200行以内か（超える場合は分割）
- [ ] コードが DRY/KISS/SOLID に従っているか

## 思考ルール

- **推測禁止**: 推測や早合点は厳禁。結論を急がない
- **複数アプローチ**: シニア開発者のように、常に複数の異なるアプローチを検討
- **シンプル優先**: 最も複雑なアプローチが必ずしも最善とは限らない
- **不必要な複雑化の回避**: 単純な問題を不必要に複雑化しない
- **根拠ベース**: データ、テスト結果、出力内容を捏造しない。すべての情報は根拠に基づく
- **90%ルール**: 90%以上の自信がないときは「わからない」旨の返答をする

## ツール活用

### Serena MCP の優先使用
- `mcp__serena__get_symbols_overview`: コード構造の把握
- `mcp__serena__find_symbol`: シンボル検索
- `mcp__serena__replace_symbol_body`: シンボル置換
- `mcp__serena__insert_after_symbol`: コード挿入
- `mcp__serena__find_referencing_symbols`: 参照元の検索

### その他
- `Read`: ファイル読み込み
- `Write`: 新規ファイル作成
- `Edit`: ファイル編集
- `Bash`: テスト実行、ビルド確認

## 一時ファイル
- 一時的に作成する必要のあるファイルは `$PROJECT_ROOT/.tmp` に配置

## 関連スキル
- `planning`: 実装前の計画立案
- `review`: 実装後のコードレビュー
