# 実装計画スキル

## 目的
実装に先立って、要件を整理し、タスクを分解し、技術的な実現可能性を検証する。

## プロセス

### 1. 要件の明確化
- ユーザーの要求を具体的な要件に落とし込む
- 曖昧な点があれば質問して明確化する
- 機能要件と非機能要件を分離する

### 2. 技術的制約の確認
- Serena MCP を使って既存のコード構造を把握
  - `mcp__serena__get_symbols_overview` でファイルの構造を確認
  - `mcp__serena__find_symbol` で関連するシンボルを検索
  - `mcp__serena__search_for_pattern` でパターン検索
- DBスキーマの確認（必要に応じて）
- 既存のアーキテクチャとの整合性を確認
- 使用する技術スタックの確認

### 3. タスク分解と優先度付け
- 1タスク = 1PR で分割可能な粒度に分解
- 各タスクの依存関係を明示
- 優先度（High/Medium/Low）を設定
- 実装内容、成果物を明記

### 4. 見積もりとリスク洗い出し
- 各タスクの実装時間を見積もる（必要に応じて）
- 技術的リスクを洗い出す
- リスクに対する対策を検討

### 5. 成果物の作成

**計画書の作成**
- `templates/plan-template.md` をベースに計画内容を作成
- 計画内容を詳細に記述して返す
- ファイル保存は**メインスレッドが実施**するため、subagent は保存不要

## 出力フォーマット
`templates/plan-template.md` を参照してフォーマットに従う。

計画内容を詳細に記述して返す。ファイル保存はメインスレッドが以下の命名規則で実施：
- `.tmp/plans/YYYY-MM-DD_<feature-name>.md`
- 例: `.tmp/plans/2025-12-18_user-authentication.md`

## 注意事項

### タスク分割の粒度
- 1タスクは **1PR** で完結する粒度
- 大きすぎる場合はさらに分割
- 小さすぎる（10行未満の変更）場合は統合を検討

### 依存関係の明示
- 前提となるタスクを明記
- 並行実施可能なタスクを明示
- ブロッカーがある場合は最優先で記載

### 技術調査の深さ
- 既存コードは必ず確認する
- 新しいライブラリ導入時は公式ドキュメントを参照
- セキュリティ、パフォーマンスへの影響を考慮

### ユーザーとのコミュニケーション
- 複数のアプローチがある場合は選択肢を提示
- 不明点は推測せず質問する
- 計画書作成後、必ずユーザーの承認を得る

## 関連ツール
- `mcp__serena__get_symbols_overview`: ファイルの構造把握
- `mcp__serena__find_symbol`: シンボル検索
- `mcp__serena__search_for_pattern`: パターン検索
- `Read`: ファイル内容の確認
- `Grep`: コード検索
- `Glob`: ファイル検索
